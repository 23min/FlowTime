# M-10.03 - MCP Dependency Pattern Enforcement

**Epic:** Dependency Constraints & Shared Resources (`work/epics/dependency-constraints/spec.md`)  
**Status:** Draft

## Goal

Enforce the **dependency pattern matrix** in the MCP server so MCP-assisted modeling emits only supported dependency patterns (Option A or Option B) and blocks unsupported behaviors (Plans 3–5) with clear guidance.

## Problem Statement

MCP modeling currently allows users (or agents) to describe dependencies in many ways, which risks producing ambiguous or incorrect models. We need MCP to enforce **canonical patterns** that match engine capabilities, and to refuse unsupported features (e.g., feedback loops, dependency retries) with actionable alternatives.

## Scope

### In Scope
- Add a **dependency pattern selector** in MCP modeling that:
  - maps user intent to Option A or Option B,
  - refuses unsupported patterns with a clear explanation,
  - emits only canonical helper output.
- Harden MCP validation so engine **info warnings** about dependency contracts become **hard errors** during MCP model generation.
- Add tests verifying:
  - correct pattern selection,
  - rejection of unsupported behaviors,
  - deterministic helper output.

### Out of Scope
- Implementing Plans 3–5 (resource pooling, compiler expansion, feedback loops).
- Changes to engine behavior or flow latency logic.

## Requirements

### Dependency Pattern Matrix
MCP must implement the matrix documented in `work/epics/completed/ai/mcp-modeling.md`:

| User intent | MCP pattern | Supported now |
| --- | --- | --- |
| Dependency in series | Option A helper | ✅ |
| Dependency as constraint | Option B helper | ✅ |
| Dependency retries/feedback loops | Plan 4/5 | ❌ |

### Canonical Output Contracts
- **Option A helper**
  - Dependency node with arrivals/served/errors
  - Effort edge with attempts volume
  - Upstream retry pressure derived (no cycles)
  - Derived series tagged with `origin: derived`
- **Option B helper**
  - Constraint registry + node references
  - Deterministic allocation rule (as in M‑10.02)
  - No retry modeling

### Validation
MCP must treat the following engine warnings as **hard errors** for model generation:
- `missing_dependency_arrivals`
- `missing_dependency_served`
- `dependency_missing_effort_edges`
- `dependency_retry_pressure_missing`

## Deliverables

### MCP Server
- Implement dependency pattern selection logic.
- Enforce strict validation mapping for dependency warnings.
- Provide clear error messaging with suggested alternatives.

### Tests
- Unit tests for pattern selection outcomes.
- Validation tests that convert dependency warnings into MCP errors.
- Golden/fixture tests for helper output (Option A/Option B).

### Documentation
- Update MCP modeling docs to reference enforcement behavior (already documented in `work/epics/completed/ai/mcp-modeling.md`).
- Add release note for M‑10.03.

- MCP emits only Option A or Option B dependency patterns.
- Unsupported dependency behaviors are rejected with actionable guidance.
- Dependency warnings are treated as hard errors in MCP modeling workflows.
- Tests confirm deterministic output and rejection behavior.
