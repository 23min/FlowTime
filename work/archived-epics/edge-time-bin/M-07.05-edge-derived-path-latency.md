# M-07.05 - Edge-Derived Path Latency (Sink Flow Latency)

**Status:** ✅ Complete  
**Dependencies:** ✅ M-07.01 (Edge Time Bin Foundations), ✅ M-07.02 (Conservation and Quality Signals), ✅ M-07.03 (Edge Time Bin UI Overlays), ✅ M-07.04 (Edge Semantics Contract)  
**Target:** Provide a consistent, engine-derived end-to-end latency series at sinks using edge flow facts and node delay contributions.

---

## Overview

This milestone introduces a **derived path latency** series for sinks (and optionally for intermediate nodes) so the UI can display end-to-end latency even when the sink has no native service time or queue metrics. The approach is explicitly **derived** from existing node metrics and **engine-emitted edge metrics** (M-07.04) rather than a full path mining or event simulation system.

### Strategic Context
- **Motivation:** Sinks represent completed work; users expect a flow latency number at the end of the process. Today, sinks lack a reliable flow latency series.
- **Impact:** Adds a stable, explainable latency series for sinks and edges, improving end-to-end observability.
- **Guardrail:** This is not full path analytics. It is a constrained, derived metric that remains consistent with the DAG evaluation model.

## Scope

### In Scope ✅
1. **Derived path latency series** for sinks based on explicit edge flow totals and node delay contributions.
2. **Schema support** for the derived latency series in time-travel state.
3. **Provenance annotations** to clarify that the latency is derived from upstream flows and node delays.
4. **UI support** to display the sink latency series and sparkline when available.

### Out of Scope ❌
- Full path mining or multi-path distribution analytics.
- Edge-level behavior modeling (capacity, queues, schedules, delays). Transit behavior must be modeled as nodes.
- Telemetry ingestion of path-level latencies (handled in Telemetry Ingestion epic).

## Requirements

### Functional Requirements

#### FR1: Derived sink latency series
**Description:** Compute a sink-level latency series per bin using upstream node delay contributions and edge flow weighting.

**Acceptance Criteria:**
- [x] A sink receives a new latency series (e.g., `pathLatencyMs`) when upstream flow and latency components exist.
- [x] The series is derived from **node-level delay components** (queue latency, service time, etc.) and **explicit edge flow totals** emitted by the engine.
- [x] The series is **per-bin** and aligns with the time grid (same bin index and window slicing behavior).

#### FR2: Provenance and semantics
**Description:** Mark the derived series clearly for UI and API consumers.

**Acceptance Criteria:**
- [x] Provenance marks the series as `origin: derived` and `aggregation: avg` (or other declared aggregation).
- [x] The meaning string explicitly states that the latency is derived from upstream flow and node delays.

#### FR3: UI visibility
**Description:** Sink nodes surface the derived latency in inspector properties and sparklines when present.

**Acceptance Criteria:**
- [x] The sink inspector shows the derived latency series with a clear label (e.g., “Flow latency (derived)”).
- [x] Sparkline uses the same data as the label (no mismatches).

### Non-Functional Requirements

#### NFR1: Deterministic output
**Target:** Derived latency is deterministic for a given run artifact.
**Validation:** Golden runs reproduce the same latency series.

#### NFR2: Backward compatibility
**Target:** Clients without this series continue to work.
**Validation:** Series is optional; absence must not break existing UI.
