# M-07.03 - Edge Time Bin UI Overlays and UX

**Status:** ✅ Complete  
**Dependencies:** ✅ M-07.01 (Edge Time Bin Foundations), ✅ M-07.02 (Conservation and Quality Signals)  
**Target:** Use edge time bins in the UI for overlays and tooltips, with clear fallback indicators when edge data is missing or approximate.

---

## Overview

This milestone makes edge metrics visible in the Time Travel UI. When edge data is available, overlays and tooltips should be driven by EdgeTimeBins. When edge data is missing or approximate, the UI falls back to node-derived overlays and clearly communicates fidelity.

### Strategic Context
- **Motivation:** Edge flows must be visible to make routing, retries, and multi-parent flows understandable.
- **Impact:** Edge overlays become truthful for simulation runs and any telemetry with edge metrics.
- **Dependencies:** Requires edge data and quality signals from M-07.01 and M-07.02.

## Scope

### In Scope ✅
1. Edge overlays use EdgeTimeBins for thickness/color when available.
2. Edge tooltips show per-edge flow totals and per-class breakdowns.
3. Fallback to existing node-derived overlays when edge data is missing, with explicit fidelity messaging.
4. Class filter applies to edge overlays when per-class edge data is available.

### Out of Scope ❌
- New path analytics or multi-edge route views.
- Edge editing or authoring tools.
- Telemetry ingestion changes.

## Requirements

### Functional Requirements

#### FR1: Edge overlays driven by edge data
**Description:** When edge metrics are present, overlays must use them instead of node-derived heuristics.

**Acceptance Criteria:**
- [ ] Edge thickness scales by `flowVolume` or `flowByClass` for the selected class.
- [ ] Edge color reflects error/retry intensity when those metrics are present.
- [ ] Overlay logic switches to edge metrics automatically when available.

#### FR2: Tooltips and labels
**Description:** Edge tooltips expose the underlying edge metrics.

**Acceptance Criteria:**
- [ ] Tooltips show `flowVolume` and class breakdowns when available.
- [ ] If edge quality is `approx` or `missing`, tooltip includes a short fidelity note.

#### FR3: Fallback behavior
**Description:** When edge data is missing or approximate, fall back to node-derived overlays with clear messaging.

**Acceptance Criteria:**
- [ ] UI shows a clear indicator (e.g., badge or caption) when overlays are approximated.
- [ ] Class filters still apply, but only where per-class edge data exists.

### Non-Functional Requirements

#### NFR1: Visual parity
**Target:** Edge overlay styles remain consistent with existing topology styling.
**Validation:** UI snapshots or render tests match existing style guidelines.
