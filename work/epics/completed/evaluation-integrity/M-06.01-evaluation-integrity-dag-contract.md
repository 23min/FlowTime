# M-06.01 — Evaluation Integrity (DAG/Spreadsheet Contract)

**Status:** ✅ Complete  
**Dependencies:** ✅ FT-M-05.13.1 (class propagation gap fix)  
**Target:** Make the engine evaluation order and class propagation strictly DAG-consistent, so post-evaluation mutation is impossible and results are always trustworthy.

---

## Overview

FlowTime models are a spreadsheet: each series is a cell, formulas depend on upstream cells, and evaluation must follow the DAG order. This contract is non-negotiable. If we ever change a "cell" after evaluation (even for class propagation), we violate the DAG and can silently invalidate downstream series.

Recent fixes exposed a gap: class series were injected after the topological pass (post-eval mutation), and downstream expression nodes were not always recomputed. This is unacceptable. The engine must enforce a single evaluation pipeline where overrides are applied before evaluation and no mutations are allowed after the DAG is computed.

This milestone establishes hard guardrails, restructures class propagation to use the same evaluation pipeline as totals, and adds tests/diagnostics that fail fast if post-eval mutation is attempted.

**Purity scope:** We keep the user-facing node model intact while compiling internal nodes into a strict DAG. This milestone removes post-eval injections for current behaviors (queue depth, retry echo, class overrides) by moving them into the compiled DAG evaluation. Templates do not change; the engine expands nodes internally. Full node compiler extraction is tracked in `work/epics/completed/evaluation-integrity/M-06.02-model-compiler.md`.

## Scope

### In Scope ✅
1. Formalize the **DAG/Spreadsheet evaluation contract** for all series, including class series.
2. Refactor class contribution evaluation to be **topologically ordered and immutable** after evaluation.
3. Add **hard guardrails** that prevent post-eval mutation (error or explicit recompute path).
4. Add **tests and diagnostics** that detect any propagation gaps or order violations.
5. Document the evaluation contract and the strict pipeline guarantees.

### Out of Scope ❌
- UI changes beyond surfacing existing warnings.
- New routing logic or schema changes unrelated to evaluation ordering.
- Performance optimizations that alter the evaluation semantics.

## Requirements

### FR1: Single Evaluation Pipeline (No Post-Eval Mutation)
**Description:** All overrides (router, queue outflow, topology semantics) must be applied before class series evaluation completes.

**Acceptance Criteria**
- [ ] Class series are computed in a single DAG-ordered pass with overrides applied up front.
- [ ] Post-evaluation mutation is disallowed (throws or emits a hard warning).
- [ ] Any override path either triggers a full recompute or is part of the pre-eval pipeline.

### FR2: Class Propagation Uses DAG Order
**Description:** Derived series (e.g., served/arrivals aliases) must inherit class series through DAG evaluation, not manual injection.

**Acceptance Criteria**
- [ ] If a series references a classed input, its class series exists unless explicitly suppressed.
- [ ] Class series for derived nodes match totals (normalized and consistent).
- [ ] Downstream expressions always recompute when upstream class series change.

### FR3: Order Violation Guardrails
**Description:** Detect and prevent any evaluation order violations.

**Acceptance Criteria**
- [ ] Invariant analysis flags missing/partial class coverage in topology semantics.
- [ ] A new guard detects "late injection" attempts and fails fast.
- [ ] Diagnostics include clear remediation guidance (fix model or pipeline).

### FR4: Test Coverage for DAG Integrity
**Description:** Tests must lock the contract and catch any regression.

**Acceptance Criteria**
- [ ] Core tests assert that overrides never leave downstream class series empty.
- [ ] Tests fail if class propagation occurs enforceably after evaluation.
- [ ] Regression tests cover serviceWithBuffer + router + derived expr chains.

## Implementation Plan

### Phase 1 — Contract + Design
1. Document the evaluation contract (DAG/Spreadsheet rules).
2. Identify all override paths that currently mutate after evaluation.
3. Define a single pipeline that applies overrides before evaluation.

### Phase 2 — Engine Refactor
1. Centralize class series evaluation in one DAG-ordered pipeline.
2. Remove or forbid post-eval injection paths.
3. Ensure router/serviceWithBuffer/topology semantics are evaluated in the same order as totals.

### Phase 3 — Guardrails + Diagnostics
1. Add "late mutation" detection (exception or high-severity warning).
2. Extend invariant warnings to cover topology semantics coverage gaps.
3. Ensure warnings surface in run manifests and UI.

### Phase 4 — Tests + Validation
1. Add DAG integrity regression tests (core + integration).
2. Add tests for derived series propagation (serviceWithBuffer -> service -> sink).
3. Run full test suite.

## Notes

This milestone is about **trust**. The engine must behave like a spreadsheet: inputs flow forward, and nothing can change after evaluation. All derived class data must be computed by the same DAG rules as totals. Any deviation should be a hard failure, not a silent mismatch.
