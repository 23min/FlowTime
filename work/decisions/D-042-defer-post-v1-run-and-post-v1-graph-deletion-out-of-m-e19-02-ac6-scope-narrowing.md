---

**Status:** active
**Context:** M-024 A5/matrix scheduled `POST /v1/run` and `POST /v1/graph` for deletion in M-025 on the premise that they are not used by first-party UIs. The audit was route-level and UI-call-site-level; it did not inspect test-file coupling. During M-025 implementation (2026-04-08), grep showed 50 `POST /v1/run` call sites and 2 `POST /v1/graph` call sites across 9 Engine test files (`ProvenanceHashTests`, `ProvenanceHeaderTests`, `ProvenanceEmbeddedTests`, `ProvenancePrecedenceTests`, `ProvenanceStorageTests`, `ProvenanceQueryTests`, `Legacy/ApiIntegrationTests`, `ParityTests`, `CliApiParityTests`). These are the primary run-creation mechanism for Engine-side runtime provenance, CLI ↔ API parity, and legacy integration coverage — directly adjacent to the E-16 purity work that E-19 is built on. Deleting the routes in this milestone would either regress ~50 provenance/parity tests (forward-only test deletion — unacceptable coverage loss) or pull substantial test-migration work that is out of scope for a runtime-cleanup milestone.
**Decision:** Scope M-025 AC6 down to deleting `GET /v1/debug/scan-directory/{dirName}` only (a genuine zombie route with no callers). Defer `POST /v1/run` and `POST /v1/graph` deletion out of M-025. Retain them as a transitional test-infrastructure surface until the Provenance/Parity/Legacy test suites are migrated to an alternative run-creation path — either a test-only in-process adapter over `Graph.Evaluate` / `RunOrchestrationService` or the supported Sim orchestration endpoint with template fixtures. Update the supported-surfaces matrix row from `delete m-E19-02` to `transitional / deferred (see work/gaps.md)`. Record the deferral and resolution path in `work/gaps.md`. No 410-style rejection stub is added now or later — when deletion eventually happens, it is forward-only per D-039.
**Consequences:** M-025 ships with 9 ACs executed and AC6 explicitly narrowed; the milestone does not reach `delete` status on `/v1/run` and `/v1/graph`. A follow-up unit of work (candidate name `m-E19-02a-engine-runtime-route-retirement`) must land the test migration before the routes can be removed. This decision also documents a process learning: the M-024 audit was route-level and UI-call-site-level, which is insufficient for runtime-cleanup scope locking — future inventory milestones should include a test-file sweep when an audit row implies route deletion. No new callers should land on `/v1/run` or `/v1/graph` in the interim; new provenance tests should use an in-process path rather than HTTP.
