---
id: D-047
title: Additive `trace` field on `/v1/goal-seek` and `/v1/optimize`
status: accepted
---

**Status:** active
**Context:** E-21 epic constraints (Scope / Constraints) restrict UI-side backend work to read-only run-adjacent endpoints per D-046; compute-endpoint shape changes require their own decision. The `/v1/goal-seek` and `/v1/optimize` endpoints already compute per-iteration state (bisection midpoints; Nelder-Mead per-iteration best vertex) and discard it before returning. The Svelte analysis surface needs that state to show users the shape of the search, not just the final number. M-011's .NET Time Machine CLI will want the same information in JSON mode.
**Decision:** Add an additive `trace` field to both response shapes. For `/v1/goal-seek`, the trace is an ordered list of `{ iteration, paramValue, metricMean, searchLo, searchHi }` entries — two `iteration: 0` boundary evaluations followed by one entry per bisection step with the post-step bracket; when the target is not bracketed the trace contains only the two boundary entries. For `/v1/optimize`, the trace is an ordered list of `{ iteration, paramValues, metricMean }` entries recorded after each per-iteration sort so `paramValues` is the current best vertex and `metricMean` is its unsigned mean (the internal minimize-sign flip is reversed at record time for maximize runs); `iteration: 0` is the initial simplex's best vertex after the pre-loop sort, `iteration: 1..N` are the post-iteration bests. Request shapes, validation rules, 400/503 behaviour, and the existing `paramValue(s)` / `achievedMetricMean` / `converged` / `iterations` fields remain unchanged. The trace captures per-iteration best state only — not the raw per-evaluation probe log.
**Consequences:** M-041 implements the trace plumbing through `GoalSeeker`, `Optimizer`, and the endpoint response records; existing API tests gain trace-shape assertions; the .NET CLI passthroughs inherit the field for free via the existing JSON pipe-through. Backwards compatibility: additive only — existing consumers that ignore unknown fields continue to work. The E-21 epic spec Scope / Constraints section is updated to reference this decision alongside D-046 as the two explicit carve-outs from "Svelte UI is a pure consumer."
**Out of scope of this decision:** the per-evaluation probe log (reflection / expansion / contraction / shrink intermediate vertices); any change to `/v1/sweep`, `/v1/sensitivity`, or other compute endpoints.
