title: Testing-rigor approach — Phase 2 baseline canary first; full golden-output canon deferred
status: accepted
---

**Status:** active
**Context:** During M-045 dogfooding the user observed three `edge_flow_mismatch_incoming` warnings on the previously-clean `transportation-basic` template. Two gaps were filed: (1) the regression itself, and (2) a structural argument that surveyed-output-only canaries cannot detect drift and we need deterministic golden-output assertions before E-22 Model Fit. The `patch/edge-flow-mismatch` investigation diagnosed (1) as **improved analyser coverage surfacing a latent template inconsistency**, not an engine bug — pre-E-24 the engine wasn't emitting per-edge `flowVolume` series so the analyser's conservation check silently skipped; post-E-24 the series is emitted, the check runs, and finds real divergence between expr-layer arrivals (`hub_dispatch * splitAirport`) and edge-layer flows (uniform-1/3 from `weight: 1` edges). 8+ shipped templates affected; `transportation-basic-classes` worst at 8 run-warnings. The patch left the underlying template-vs-engine authority question open for a future engine milestone; the question is which side is canonical for edge flow volumes (expr nodes encode splits, topology edge weights would need to mirror them).

**Decision:** Adopt the **Phase 2 baseline canary** as the testing-rigor approach for now. Defer full per-template golden-output pinning until either (a) the engine truth-gate work picks up the canary's findings as scoped milestones, or (b) the gate proves insufficient at catching a class of drift. Concretely: extend `tests/FlowTime.Integration.Tests/TemplateWarningSurveyTests.cs::Survey_Templates_For_Warnings` Phase 2 (live-API survey) to hard-assert per-template `run-warn` counts against an `ExpectedRunWarnings` baseline dictionary maintained in the test source. The dictionary's xmldoc names accepted-for-now non-zero baselines with rationale pointers to `work/gaps.md`. Drift in either direction (regression upward, or sanctioned-fix downward without baseline update) fails the build with a clear before/after message. The Phase 1 (in-process tier-3) `val-err == 0` hard assertion from E-24 M-053 is retained.

**Why baseline-canary, not full-golden-canon:** the regression that motivated the gap had a single specific failure shape — *count of run-warnings per template diverged from previously-observed count*. A baseline canary catches that exact shape with one dictionary edit per template and a few lines of assertion code; a full golden-output canon (per-series per-bin numeric pinning + warning-set pinning + manifest-field pinning + tolerance comparison + regenerate workflow) is a milestone-scale investment that buys broader coverage but at a per-deliberate-change tax. The patch deliberately scoped to the lighter intervention because the intervention sufficed for the observed failure mode and unblocks E-15 / E-22 sequencing. If a future regression slips past the baseline gate (e.g. same warning count but different warning content; or numeric drift inside a stable warning count), that is the empirical signal to invest in full canon. Until then, the gate is enough.

**Consequences:** The `Survey_Templates_For_Warnings` canary now actively prevents the class of silent drift that produced the M-045 finding — any future change that bumps any template's run-warning count above its baseline (or below without a deliberate baseline edit) fails CI. The 9 known-non-zero baselines locked in 2026-05-01 (`transportation-basic`, `transportation-basic-classes`, `it-system-microservices`, `manufacturing-line`, `network-reliability`, `supply-chain-incident-retry`, `supply-chain-multi-tier`, `supply-chain-multi-tier-classes`, `warehouse-picker-waves`) trace back to the edge-flow-mismatch investigation; resolving the underlying expr-vs-edge-weight authority question lowers them all to zero in the same commit that makes the design call. **E-15 Telemetry Ingestion is no longer testing-rigor-blocked** — the baseline canary is the gate that the gap entry argued was needed. **E-22 Model Fit may still want full golden-output canon** when it lands (model-fit accuracy depends on detectable engine-output stability at numeric precision, which baseline-counts don't catch); that decision lives at E-22 plan time. The two gaps filed in M-045 dogfooding remain open in `work/gaps.md`: the transportation-basic gap retains the design question and the affected-template list; the testing-rigor gap is materially answered by this decision but not yet closed (full canon is still future scope when warranted). Patch landed: `patch/edge-flow-mismatch`, single PR back to `main`.
