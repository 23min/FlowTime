---
id: D-048
title: E-23 Model Validation Consolidation — delete `ModelValidator`, do not delegate
status: accepted
---

**Status:** active
**Context:** Mid-session during M-044 Validation Surface scoping, an evidence-gathering survey (test at `tests/FlowTime.Integration.Tests/TemplateWarningSurveyTests.cs`) discovered that every one of the twelve current templates produces tier-3 validation errors (`/grid/start: All values fail against the false schema`) under `TimeMachineValidator.Validate(..., ValidationTier.Analyse)` while every template also runs successfully under `POST /v1/run`. Root cause: two validators with overlapping scope silently disagreeing — `ModelValidator.Validate` (hand-rolled, imperative, uses `.IgnoreUnmatchedProperties()`, does not consult the schema; called from `POST /v1/run`) and `ModelSchemaValidator.Validate` (reads `docs/schemas/model.schema.yaml` and runs full JSON-schema evaluation; called from `POST /v1/validate` via `TimeMachineValidator`). Git history shows `ModelValidator` landed in `6432057 feat(core): implement schema validation (M2.9 Phase 2 Step 2)` as the pre-schema home-rolled checker; `ModelSchemaValidator` was added later once the JSON schema existed. The migration to schema-first was never completed — the two validators coexist, and the schema forbids `grid.start` which Sim deliberately emits (`SimModelBuilder.cs:34-36` copies `window.start` into `grid.start`) and the Engine's `ModelParser.cs:62` consumes (Blazor's time-travel UX and Svelte's `StateWindowResponse.timestampsUtc` both depend on this chain, so removing `grid.start` from Sim would regress both UIs). The failure mode is strictly worse than "two schemas drift apart" — one authoritative schema exists, but a parallel hand-rolled validator covers the same contract surface and silently disagrees. This directly violates the Truth-Discipline guard in `.ai-repo/rules/project.md`: *"Do not restate a canonical contract in many places from memory — point to the owning doc."*
**Decision:** Promote the cleanup to a first-class mini-epic **E-23 Model Validation Consolidation** with three milestones (M-046 Schema Alignment, M-047 Call-Site Migration, M-048 Delete `ModelValidator`). Endpoint: `ModelSchemaValidator.Validate` is the **only** validator in the codebase for model YAML. `ModelValidator` is **deleted outright**, not delegated — per user instruction and the 2026-04-23 "API stability does not mean keep old functions around" guard. Every rule currently enforced by `ModelValidator` (schemaVersion const, legacy-field rejection for top-level `arrivals`/`route`, required `grid`, etc.) is expressed declaratively in `docs/schemas/model.schema.yaml`. Schema is updated to allow `grid.start` as an optional ISO-8601 string so it matches runtime reality. `SimModelBuilder`'s `window.start → grid.start` copy is unchanged (Sim is correct as-is). E-21 Svelte Workbench is paused at M-043 (complete on milestone branch, merge-to-epic deferred) to prioritise E-23 — M-044 Validation Surface becomes the reentry point and consumes the consolidated validator. Survey test is committed to `main` as a bootstrap canary BEFORE E-23 milestones start so the test history predates the cleanup and the red-to-green transition is visible across the epic's commit series. Sibling-checkout grep of `/workspaces/flowtime-sim-vnext` for `ModelValidator` references returned zero hits — deletion is safe for the sister repo. In-repo grep shows 85 `ModelValidator` references across `*.cs` / `*.md` files that M-047 will migrate. Rejected alternative: delegation (`ModelValidator.Validate` wraps `ModelSchemaValidator.Validate`) — fails the "API stability ≠ keep old functions around" guard and preserves the shape of the coexistence problem.
**Consequences:** E-21 pauses; M-044 Validation Surface cannot plan active pre-run validation until E-23 lands, but the passive run-warning path it already had is unaffected. All 85 in-repo `ModelValidator` call sites migrate to `ModelSchemaValidator` in M-047. The `docs/schemas/model.schema.yaml` file becomes load-bearing for every validation decision — when a rule needs changing (e.g. adding a new required field, banning a new legacy shape), it is a schema edit, not a code edit. Error-message phrasing changes slightly (`ModelSchemaValidator` produces `/path: <rule failure>` messages vs `ModelValidator`'s hand-rolled strings); tests that assert on exact messages need a one-time reconciliation in M-047. The survey test `TemplateWarningSurveyTests` is a permanent regression guard for any future validator drift — if a new validator appears, or if Sim and the schema drift again, the survey's per-template `val-err` column flags it immediately.
