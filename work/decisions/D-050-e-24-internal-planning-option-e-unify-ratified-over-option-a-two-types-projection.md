title: E-24 internal planning — Option E (unify) ratified over Option A (two types + projection)
status: accepted
---

**Status:** active
**Context:** The initial E-24 draft (D-049) scoped six milestones around **Option A**: preserve `SimModelArtifact` and `ModelDefinition` as two distinct types, introduce a projection layer (`SimModelArtifact.ToEngineSubmission()`), and let the schema describe only the submission payload. A follow-on investigation into `SimModelArtifact`'s purpose (agent `a5aa3dfe26394aff5`) established that the two-type split was accidental, not designed: `SimModelArtifact` was introduced in commit `ce9ec9e` (Oct 2025) shaped to the template schema rather than to `model.schema.yaml` (which predated it by a month and already described what the Engine consumes); `docs/schemas/README.md:34` claims parity that the code does not deliver; no C# type boundary exists between the two shapes — the only interface is a YAML string; each feature addition since has paid a duplication tax (e.g. commit `51a99b9` added classes to both types in the same commit, +23 / +28 lines). The only field with genuine dual concern is `provenance`, and both sides already agree on the keys, they just disagree on casing. Option A would have preserved the accidental split, converted the leak into a designed feature, and accepted the ongoing duplication tax.

**Decision:** Within E-24 planning, ratify **Option E: unify**. Collapse `SimModelArtifact` and `ModelDefinition` into a single unified post-substitution model type. One C# type. One YAML schema (`docs/schemas/model.schema.yaml`) describing it. One validator (`ModelSchemaValidator`) validating it on both sides. `Template` (authoring-time, pre-substitution) stays distinct — that separation is load-bearing. Forward-only — no migration of stored bundles; Sim regenerates from templates going forward. camelCase throughout per project rule. E-24's milestone count collapses from six to five:

1. **M-049 Inventory & Design Decisions** (doc-only) — unchanged in concept, refreshed to record per-field decisions under the unification framework and to resolve six open questions (unified-type home, unified-type name, `outputs[].as` semantics, `nodes[].source` forward contract, provenance shape, canary variant).
2. **M-050 Unify Model Type** (architectural, largest milestone) — replaces the prior "Engine Submission Payload Separation" milestone. Introduces the unified type, rewrites `SimModelBuilder` to emit it directly, switches Engine intake to parse it directly, deletes `SimModelArtifact` and satellites (`SimNode`, `SimOutput`, `SimProvenance`, `SimTraffic`, `SimArrival`, `SimArrivalPattern`), drops leaked-state fields from emission, regenerates fixtures.
3. **M-051 Schema Unification** (schema) — rewrites `docs/schemas/model.schema.yaml` to describe the unified type in camelCase, rewrites `docs/schemas/README.md`, audits `docs/architecture/`.
4. **M-052 Parser/Validator Scalar-Style Fix** (validator) — unchanged in concept, renumbered from M-053. Independent of unification.
5. **M-053 Canary Green + Hard Assertion** (close) — unchanged in concept, renumbered from m-E24-06. E-23 resume gate.

The old M-052 "Sim Emission Discipline" is absorbed into the new M-050 (emitter changes fall out of building the unified type directly).

**Consequences:** E-24 now has five milestones, not six. The epic spec is rewritten around Option E framing (spec.md at epic-branch tip reflects this decision). M-049 ratifies the ADRs: **ADR-E-24-01 Unify the post-substitution model type**, **ADR-E-24-02 Forward-only regeneration**, **ADR-E-24-03 Schema declares only consumed fields**, **ADR-E-24-04 (candidate) `ScalarStyle.Plain` gates numeric/boolean coercion in `ParseScalar`**. The unified type absorbs every currently-used field across both old types (minus leaked state); the final field list is named in M-049 before M-050 starts. `SimModelArtifact` and its satellite types are deleted outright in M-050 per Truth Discipline — no coexistence window, no transitional dual-type state. Forward-only stance means every test fixture and sample bundle authoring the old two-type shape is regenerated in M-050; any fixture that cannot be regenerated is deleted rather than patched. Under unification, the M-046 "architectural seam" (48 errors) and "emitter redundancy" (1 shape) classifications collapse entirely; "emitter drift" (375 errors) collapses because `outputs[]` shape is owned by the unified type's serializer; "systematic rule violation" (72 errors) collapses with the schema rewrite; "validator defect" (~231 errors) survives to M-052 as planned. E-23 pause and M-047/M-048 dependencies remain unchanged; at E-24 close both become mechanical cleanup because `ModelSchemaValidator`, the schema, Sim's emitter, and the Engine's reader will literally share a single type definition.
