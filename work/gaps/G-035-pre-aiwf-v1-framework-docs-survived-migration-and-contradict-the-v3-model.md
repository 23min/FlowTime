## What's missing

Several pre-aiwf v1 framework docs survived the v1→v3 migration on 2026-05-01 and now contradict the aiwf v3 planning model. They describe a separate-tracking-document workflow under `work/milestones/tracking/[MILESTONE-ID]-tracking.md` that does not exist in v3 (the v3 model tracks all milestone work inline in the milestone `.md` itself). They were written for the v1 `.ai/` submodule + `bash sync.sh` framework that was torn down in commit `b80d4c3`.

Concrete files contradicting v3:

- `docs/development/milestone-documentation-guide.md` — extensive v1 ritual including "Create tracking document from template `docs/development/TEMPLATE-tracking.md` → `work/milestones/tracking/[MILESTONE-ID]-tracking.md`" (lines 368, 371, 388, 449, 474, 529, 535, 605).
- `docs/development/TEMPLATE-tracking.md` — the v1 tracking-doc template itself.
- `docs/development/milestone-session-guide.md` — references `work/milestones/tracking/[CURRENT_MILESTONE]-tracking.md` (line 20).
- `docs/development/milestone-rules-quick-ref.md` — likely contains the same convention; needs audit.
- `docs/development/branching-strategy.md` — contains `work/epics/completed/ui-perf/` → branch convention from v1 (line 29) inconsistent with v3's "epic dirs stay in place at `work/epics/E-NN-<slug>/` regardless of status".
- `docs/index.md` — autogenerated index that surfaces the v1 templates as authoritative (lines 276, 278, 296, 319, 1031, 1032, 1211, 1226).
- `docs/flowtime-engine-charter.md` and `docs/modeling.md` — reference `work/epics/completed/...` paths from v1 layout.

The pre-aiwf release notes under `docs/releases/*.md` are historical and reference the v1 paths legitimately (those releases happened under v1); they should not be edited.

## Why it matters

These files are read by both human authors and AI assistants as "current project rules" because they live under `docs/development/` and `docs/`. During M-066 planning, the planner agent and builder agent both propagated the v1 "tracking doc" convention into M-066/M-067/M-068 specs because CLAUDE.md (since corrected) plus these v1 docs reinforced the false convention. Result: a sibling `m-E25-01-edge-flow-authority-tracking.md` got created and had to be merged back into the milestone spec body and deleted. Without cleanup, every future planning round will hit the same trap.

Found because the user spotted the tracking doc in the working tree and asked "where did you read about that?" — the trail led straight to `docs/development/milestone-documentation-guide.md` and `TEMPLATE-tracking.md`.

## Likely fix

A focused docs-cleanup pass — archive or delete the v1-only docs:

1. **Move** `docs/development/milestone-documentation-guide.md`, `docs/development/TEMPLATE-tracking.md`, `docs/development/milestone-session-guide.md`, `docs/development/milestone-rules-quick-ref.md` to `docs/archive/` with a short header noting "pre-aiwf v1 framework; superseded by aiwf v3 (see CLAUDE.md and `aiwf` CLI)". Or delete outright if no historical value beyond what's already preserved in git history.
2. **Audit** `docs/development/branching-strategy.md` — strip v1-isms (`work/epics/completed/...`) and align with v3 conventions (epics stay at `work/epics/E-NN-<slug>/`; status lives in frontmatter).
3. **Re-derive** `docs/index.md` from the surviving doc set so the v1 templates stop being indexed as current.
4. **Audit** `docs/flowtime-engine-charter.md` and `docs/modeling.md` for `work/epics/completed/` references that point at content that was actually relocated to `work/archived-epics/<slug>/` during the migration.

Smallest viable fix: a `wf-patch`-shaped change that does (1) and (2), leaving (3) and (4) for a follow-up if time-boxed.
