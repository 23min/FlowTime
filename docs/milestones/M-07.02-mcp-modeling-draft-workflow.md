# M-07.02 - MCP Modeling Draft Workflow

**Status:** Complete  
**Dependencies:** M-07.01 (MCP Server PoC)  
**Target:** Add draft template authoring, validation, and execution tools to the MCP server and extend Sim to support drafts without a second instance.

---

## Overview

This milestone extends the MCP server to support draft templates in a working area. It introduces tools for creating drafts, applying patches, validating templates, generating models, and running the draft end-to-end. The goal is a closed authoring loop without writing directly into `templates/`.

### Strategic Context
- **Motivation:** Enable safe, iterative template authoring through chat.
- **Impact:** Users can modify and validate templates without touching approved assets.
- **Dependencies:** Requires the PoC MCP server and stable Sim orchestration endpoints.

---

## Scope

### In Scope
1. Draft template storage under `templates-draft/`.
2. Tools to create, read, diff, and patch drafts.
3. Template validation and analyzer reporting via Sim.
4. Model generation and run execution from draft content.
5. Sim API support for drafts using a single Sim instance.
6. Guardrails to prevent writes into `templates/`.

### Out of Scope
- Promotion of drafts into `templates/`.
- UI-based template editors.
- Data intake and profile fitting tools.
- Chart digitization.

### Future Work (Optional)
- Draft promotion workflow with explicit confirmation.
- Data intake and profile fitting (M-07.03).

## Requirements

### Functional Requirements

#### FR1: Draft Creation and Storage
**Description:** Create drafts from scratch or by cloning approved templates into `templates-draft/`.

**Acceptance Criteria:**
- [ ] Drafts are stored only in `templates-draft/`.
- [ ] Draft creation records metadata (source template, timestamp).

#### FR2: Draft Patching
**Description:** Apply structured edits to draft content.

**Acceptance Criteria:**
- [ ] Patches are applied atomically.
- [ ] Concurrency protection via content hash or etag.

#### FR3: Draft Validation
**Description:** Validate draft templates using Sim analyzers.

**Acceptance Criteria:**
- [ ] Validation returns structured errors and warnings.
- [ ] Analyzer warnings are surfaced without blocking runs by default.

#### FR4: Draft Execution
**Description:** Generate a model from the draft and run via Sim and Engine.

**Acceptance Criteria:**
- [ ] Draft runs produce engine-readable bundles.
- [ ] Run identifiers are returned for inspection tools.

### Non-Functional Requirements

#### NFR1: Isolation
**Target:** Draft operations must never modify `templates/`.
**Validation:** All write operations target only `templates-draft/`.

#### NFR2: Single Sim Instance
**Target:** Draft validation/run must work without running a second Sim instance.
**Validation:** Draft operations succeed through the primary Sim API.

**Tasks:**
1. Extend Sim API with draft endpoints (`/api/v1/drafts/*`) that accept draft sources.
2. Wire validation tool to Sim analyzers using draft sources.
3. Generate models and orchestrate runs from draft content.
4. Return run identifiers for inspection tools.

### Draft Source API (Single Sim Instance)

Draft operations should be supported through dedicated draft endpoints rather than overloading approved templates.

**Endpoints (proposed):**
- `POST /api/v1/drafts/validate`
- `POST /api/v1/drafts/generate`
- `POST /api/v1/drafts/run`

**Request Body (shared shape):**
```json
{
  "source": {
    "type": "draftId | inline",
    "id": "draft-123",
    "content": "<yaml>"
  },
  "parameters": { "bins": 288 },
  "mode": "simulation",
  "actor": {
    "tenantId": "t1",
    "userId": "u1"
  }
}
```

**Notes:**
- `source.type` chooses whether Sim loads from draft storage (`draftId`) or uses inline YAML (`inline`).
- `actor` is optional; when omitted, Sim should proceed without identity enforcement and omit actor metadata.
- Draft validation should return analyzer warnings without mutating approved templates.

**Deliverables:**
- Validation and run tools wired to HTTP endpoints.

**Success Criteria:**
- [ ] Drafts can be validated and executed end-to-end using a single Sim instance.
