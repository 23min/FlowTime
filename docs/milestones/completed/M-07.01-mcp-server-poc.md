# M-07.01 - MCP Server PoC (Node/TypeScript)

**Status:** Implemented (Pending Commit)  
**Dependencies:** Stable FlowTime.Sim.Service and FlowTime API HTTP surfaces  
**Target:** Deliver a minimal MCP server that proves the chat-to-run-to-inspect loop.

---

## Overview

This milestone establishes a minimal, Node/TypeScript MCP server that can be launched from an external chat client (for example, VS Code) and can execute a complete loop: run a template, import the run into the engine, and inspect graph/state outputs. The focus is viability and speed, not production hardening.

### Strategic Context
- **Motivation:** Validate that MCP-based chat interaction is a viable workflow for FlowTime modeling and analysis.
- **Impact:** Enables the first end-to-end chat loop without touching the FlowTime UI.
- **Dependencies:** Relies on existing Sim orchestration and Engine run/state endpoints.

---

## Scope

### In Scope
1. Node/TypeScript MCP server with stdio transport.
2. Minimal tool set to run a template and inspect results.
3. HTTP integration with FlowTime.Sim.Service and FlowTime API.
4. Basic guardrails (run budgets, timeouts, max bins).
5. Configuration via environment variables.

### Out of Scope
- Draft template authoring tools.
- Data intake and profile fitting tools.
- Authentication or multi-tenant access control.
- UI integration inside FlowTime.UI.
- Chart digitization or image inputs.

### Future Work (Optional)
- Add draft authoring and validation tools (M-07.02).
- Add data intake and profile fitting tools (M-07.03).

## Requirements

### Functional Requirements

#### FR1: MCP Server Boot and Transport
**Description:** Provide a console MCP server that can be launched by an MCP client over stdio.

**Acceptance Criteria:**
- [ ] Server starts with no interactive prompts.
- [ ] Tool schemas are discoverable by the client.
- [ ] Invalid tool invocations return structured errors.

#### FR2: Run Orchestration
**Description:** Allow the client to trigger a template run via FlowTime.Sim.Service and import it into the Engine.

**Acceptance Criteria:**
- [ ] A tool call triggers Sim orchestration and returns a run identifier.
- [ ] The resulting run is imported into FlowTime API for state queries.

#### FR3: Inspection Tools
**Description:** Provide read-only tools to inspect the run.

**Acceptance Criteria:**
- [ ] Graph inspection tool returns node and edge metadata for a run.
- [ ] State window tool returns time-binned series for requested nodes and metrics.
- [ ] Run summary tool returns warnings and key metrics.

#### FR4: Guardrails
**Description:** Enforce safe defaults for run size and duration.

**Acceptance Criteria:**
- [ ] Run budget limits are enforced server-side.
- [ ] Timeouts are applied to long-running requests.

### Non-Functional Requirements

#### NFR1: Isolation
**Target:** MCP server must remain independent from FlowTime build and runtime.
**Validation:** No compile-time dependency on FlowTime projects.

#### NFR2: Configurability
**Target:** All endpoints and working paths are configured by environment variables.
**Validation:** Server boots with no hard-coded URLs or paths.

## Implementation Plan

### Phase 1: MCP Server Skeleton
**Goal:** Create the Node/TypeScript MCP server with stdio transport and configuration handling.

**Tasks:**
1. Define tool schemas for the minimal PoC tool set.
2. Implement configuration loading and validation.
3. Implement error handling and structured responses.

**Deliverables:**
- MCP server entrypoint with stdio transport.
- Tool schema registry for PoC tools.

**Success Criteria:**
- [ ] Server starts and registers tools without errors.
- [ ] Invalid inputs return structured error responses.

### Phase 2: Sim and Engine Integration
**Goal:** Wire MCP tools to FlowTime.Sim.Service and FlowTime API endpoints.

**Tasks:**
1. Implement orchestration call to Sim and handle responses.
2. Import run bundles into the Engine.
3. Implement graph/state/summary retrieval.

**Deliverables:**
- Working run -> import -> inspect loop using HTTP endpoints.

**Success Criteria:**
- [ ] A run can be triggered and inspected through MCP tools.

### Phase 3: Guardrails and Documentation
**Goal:** Enforce safety limits and document PoC usage.

**Tasks:**
1. Add run budget limits and timeouts.
2. Document environment variables and expected endpoints.

**Deliverables:**
- Guardrails enforced in code.
- Minimal usage guidance in docs.

**Success Criteria:**
- [ ] Oversized runs are rejected with clear errors.

## Test Plan

### Test-Driven Development Approach
**Strategy:** RED -> GREEN -> REFACTOR across tool boundaries.

### Test Categories

#### Unit Tests
**Focus:** Tool routing, input validation, error mapping.

**Key Test Cases:**
1. Tool invocation with missing required fields returns a structured error.
2. Tool invocation with invalid configuration fails fast.

#### Integration Tests
**Focus:** HTTP calls to Sim and Engine endpoints.

**Key Test Cases:**
1. Orchestration request returns a run bundle reference.
2. Engine import returns a run identifier that can be queried.

#### E2E Tests
**Focus:** Full chat loop viability.

**Key Test Cases:**

- **Integration Tests:** Cover the Sim orchestration and Engine import flow.
