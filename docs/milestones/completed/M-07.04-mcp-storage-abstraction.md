# M-07.04 - MCP Storage Abstraction

**Status:** In Progress  
**Dependencies:** M-07.02 (Draft Workflow), M-07.03 (Data Intake)  
**Target:** Remove direct filesystem access by introducing a pluggable storage layer for drafts, models, and run bundles.

---

## Overview

This milestone makes MCP modeling and analysis workflows storage-agnostic. Drafts, model artifacts, and run bundles must be accessed via HTTP-backed APIs or storage providers rather than local paths. This enables multi-user deployments without requiring repo filesystem access.

### Strategic Context
- **Motivation:** Production deployments cannot depend on shared local filesystem paths.
- **Impact:** Enables multi-user usage with service-managed storage.
- **Dependencies:** Requires stable MCP tool surface and draft workflow.

---

## Scope

### In Scope
1. Storage abstraction for drafts, models, and run bundles.
2. Pluggable storage backends: filesystem (dev), object storage with index file, object storage with DB index.
3. HTTP surfaces in Sim/Engine for draft and run artifact access.
4. MCP server updated to use storage references (IDs/URIs) rather than paths.
5. Configuration-driven selection of storage backend.

### Out of Scope
- Cloud-specific deployment details (Azure/AWS/GCP).
- Auth and multi-tenant policy design beyond basic scoping.
- UI changes.

---

## Requirements

### Functional Requirements

#### FR1: Draft Storage Abstraction
**Description:** Drafts are created, updated, and read through a storage abstraction rather than direct filesystem access.

**Acceptance Criteria:**
- [ ] MCP server does not write directly to `templates-draft/`.
- [ ] Draft operations use storage identifiers (draftId) and return content hashes.

#### FR2: Run Bundle Storage Abstraction
**Description:** Run bundles are stored and referenced via storage identifiers or URIs.

**Acceptance Criteria:**
- [ ] Engine can import run bundles via storage references (not local paths).
- [ ] Run artifacts can be retrieved for analyst tools using storage references.

#### FR3: Pluggable Backends
**Description:** Storage backend is selected by configuration.

**Acceptance Criteria:**
- [ ] Filesystem backend available for local dev.
- [ ] Object storage backend with index file supported.
- [ ] Optional database-backed index supported.

### Non-Functional Requirements

#### NFR1: Deployment Readiness
**Target:** MCP server, Sim, and Engine can run without shared filesystem access.
**Validation:** No local path assumptions in API contracts.

## Implementation Plan

### Phase 1: Storage Contracts
**Goal:** Define storage interfaces and configuration.

**Tasks:**
1. Define storage abstractions for drafts/models/runs.
2. Define storage references (IDs/URIs) and hash/etag rules.
3. Add configuration for backend selection.

**Success Criteria:**
- [ ] Interfaces defined and configuration documented.

### Phase 2: Sim + Engine Storage Surfaces
**Goal:** Expose HTTP APIs that use storage references.

**Tasks:**
1. Add draft CRUD and validation endpoints that store content in the chosen backend.
2. Update model generation to return storage references.
3. Update run import to accept storage references for bundles.

**Success Criteria:**
- [ ] Draft validate/generate/run flows work without local paths.

### Phase 3: MCP Server Refactor
**Goal:** Use storage references everywhere.

**Tasks:**
1. Replace filesystem draft store with HTTP calls.
2. Update run tools to pass storage references to Engine.
3. Update tests to run against storage-backed endpoints.

**Success Criteria:**
- [ ] End-to-end modeling loop works without repo access.
