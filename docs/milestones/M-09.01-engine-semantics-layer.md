# M-09.01 - Engine Semantics Layer (Contract Hardening)

**Status:** Planned  
**Dependencies:** M-06.01/M-06.02 (Evaluation Integrity), M-07.01–M-07.06 (Edge Time Bins), M-08.01–M-08.05 (MCP pass-through)  
**Target:** Formalize FlowTime Engine as the semantics layer with stable, testable contracts for external consumers.

---

## Overview

The engine already computes derived metrics and warnings for `/state`, `/state_window`, `/graph`, and `/metrics`. What is missing is a **complete, authoritative semantics contract** that tells consumers exactly what each series means, when it is present, and how to interpret nulls or warnings. This milestone hardens those semantics into explicit, stable contracts that remain consistent across simulation and telemetry-derived bundles. The goal is to make the engine a reusable semantics layer for UIs, BI tools, and MCP clients—without client-side re-derivation or implicit assumptions.

**Why this matters even though we already have APIs/schemas:**\n- We currently validate shapes, but do not lock down the **meaning** of derived values in golden tests.\n- Series semantics (origin/aggregation/gating) are documented in multiple places but not as a single authoritative catalog.\n- Warning codes exist, but their triggers and interpretation are not consolidated or versioned.\n- External consumers (MCP/BI) need a clearly documented “pass-through” contract to avoid client-side logic drift.

### Strategic Context
- **Motivation:** External consumers must trust engine outputs and avoid re-implementing logic.
- **Impact:** Clear, stable contracts with validated outputs and explicit warnings.
- **Dependencies:** Relies on compile-to-DAG integrity and edge semantics already shipped.

## Requirements

### Functional Requirements

#### FR1: Contract Semantics Catalog
**Description:** Define a precise semantics catalog for node and edge series produced by the engine.

**Acceptance Criteria:**
- [ ] Each engine-exposed series documents units, aggregation, origin (explicit/derived), and gating rules.
- [ ] Catalog covers all **node series** exposed in `/state` and `/state_window` (e.g., arrivals, served, capacity, utilization, queue, latencyMinutes, serviceTimeMs, flowLatencyMs, attempts, retry volume/total volume, SLA series, throughput ratio, etc.).
- [ ] Catalog covers all **edge series** exposed by edge metrics (flow/attempt/retry volumes, totals, quality flags).
- [ ] Series semantics are aligned across API responses, schema docs, and UI provenance.
- [ ] Engine consumers (UI/MCP/BI) can rely on these semantics without local derivations.

#### FR2: Stable Contract Outputs
**Description:** The engine outputs stable, deterministic `/state` and `/state_window` responses for canonical bundles.

**Acceptance Criteria:**
- [ ] Golden tests lock down representative outputs for key node kinds and edge metrics.
- [ ] Derived series are present only when inputs exist; missing inputs yield explicit nulls + warnings.
- [ ] Edge metrics and quality flags are consistently present when available.

#### FR3: Warning Governance
**Description:** Warnings are first-class semantics, not UI heuristics.

**Acceptance Criteria:**
- [ ] Invariant and validation warnings are emitted from the engine and persisted in run artifacts.
- [ ] Warning messages and codes are documented and stable.
- [ ] At minimum, document currently observed warning families (e.g., queue depth mismatch, missing series, edge flow mismatch, class mismatch, edge behavior violations/lag).
- [ ] UI/MCP consume warnings as-is without re-interpretation.

#### FR4: External Consumer Contract
**Description:** Clearly define what external tools should and should not compute.

**Acceptance Criteria:**
- [ ] Engine semantics doc explicitly lists what is authoritative (derived metrics, quality flags, warnings).
- [ ] Consumers are instructed to avoid recomputation and only pass through.

### Non-Functional Requirements

#### NFR1: Determinism
**Target:** Same bundle inputs produce identical outputs across runs.

#### NFR2: Schema Discipline
**Target:** Contracts remain aligned with `docs/schemas/` and no drift is introduced.

## Implementation Plan

### Phase 1: Semantics Inventory + Docs
**Goal:** Enumerate and document current engine semantics for node and edge series.

**Tasks:**
1. Build a semantics catalog for node and edge series (units, origin, aggregation, gating).\n   - Explicitly list which series are **derived** vs **explicit**.\n   - Capture gating rules (e.g., queue latency gate closed, missing servedCount).\n2. Align `docs/architecture/engine-semantics-layer/README.md` with the catalog.\n3. Cross-check API schema docs (`docs/schemas/`), UI provenance docs, and metric catalog code.\n4. Identify any inconsistencies or missing series metadata and record fixes.

**Success Criteria:**
- [ ] Semantics catalog exists and is referenced from engine semantics docs.
- [ ] Schema docs and provenance docs are aligned.

### Phase 2: Contract Hardening Tests
**Goal:** Add regression coverage for semantic outputs (meaning, not just shape).

**Tasks:**
1. Add golden tests for `/state` and `/state_window` covering:\n   - service, serviceWithBuffer, router, sink\n   - edge metrics (flow/attempt/retry volumes, quality)\n2. Add tests for warning emission and missing-input behavior.\n3. Verify metadata fields (origin, aggregation) are exposed and consistent.\n4. Add at least one golden that asserts a derived series is **null + warning** when inputs are missing, to prevent silent fallbacks.

**Success Criteria:**
- [ ] Tests fail on semantic drift.

### Phase 3: Consumer Contract Notes
**Goal:** Make engine outputs explicitly authoritative.

**Tasks:**
1. Document which fields are derived vs explicit in engine outputs and how to interpret nulls.\n2. Update MCP and UI docs with "pass-through only" guidance and examples.\n3. Document how to interpret warnings and quality flags (e.g., when to treat values as inferred vs authoritative).

**Success Criteria:**
- [ ] Consumers are clearly instructed to avoid re-derivation.
