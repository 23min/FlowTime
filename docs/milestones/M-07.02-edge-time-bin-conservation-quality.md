# M-07.02 - Edge Time Bin Conservation and Quality Signals

**Status:** ✅ Complete  
**Dependencies:** ✅ M-07.01 (Edge Time Bin Foundations)  
**Target:** Add soft conservation checks and explicit quality signals so edge data can be trusted and debugged.

---

## Overview

With EdgeTimeBins in place, we need systematic validation that edge flows align with node metrics, and a clear signal when edge data is exact, approximate, or missing. This milestone adds conservation analyzers and quality flags that the API and UI can surface.

### Strategic Context
- **Motivation:** Edge data is only useful if it is consistent and its fidelity is explicit.
- **Impact:** Users and tools can see conservation warnings and edge-data quality in run diagnostics and state responses.
- **Dependencies:** Requires edge metrics emitted in M-07.01.

## Scope

### In Scope ✅
1. Node-edge conservation analyzers (per bin, per class where available).
2. Edge data quality flags (exact/approx/missing/partial) surfaced in state and run diagnostics.
3. Warning codes for conservation drift and partial class coverage.

### Out of Scope ❌
- UI overlay changes (handled in M-07.03).
- Telemetry ingestion for edges (handled in Telemetry Ingestion epic).
- Hard enforcement that fails runs (warnings only in this milestone).

## Requirements

### Functional Requirements

#### FR1: Node-edge conservation analyzer
**Description:** Verify that node served/arrivals match edge sums within tolerance.

**Acceptance Criteria:**
- [x] Analyzer checks outgoing edge flow vs node served per bin.
- [x] Analyzer checks incoming edge flow vs node arrivals per bin.
- [x] When class data is present, per-class conservation is checked.
- [x] Deviations beyond tolerance emit warnings (not errors).

**Examples:**
- `edge_flow_mismatch_outgoing`: served != sum(outgoing edge flows)
- `edge_flow_mismatch_incoming`: arrivals != sum(incoming edge flows)
- `edge_class_mismatch`: per-class totals do not reconcile

#### FR2: Edge data quality flags
**Description:** Provide explicit quality metadata for edge metrics.

**Acceptance Criteria:**
- [x] State responses include an `edgeQuality` (or equivalent) indicator per run.
- [x] Quality values include `exact`, `approx`, `missing`, `partialClass`.
- [x] Edge quality is derived from the presence of edge series and class coverage.

#### FR3: Diagnostics integration
**Description:** Surface conservation warnings alongside other run warnings.

**Acceptance Criteria:**
- [x] `/v1/runs` diagnostics include edge conservation warnings.
- [x] Warnings include node id, edge ids (where applicable), and bin range.

### Non-Functional Requirements

#### NFR1: Low overhead
**Target:** Conservation checks should not materially change run time for typical template sizes.
**Validation:** Performance tests show no significant regression beyond existing tolerances.

**Goal:** Add conservation checks for totals.

**Success Criteria:**
