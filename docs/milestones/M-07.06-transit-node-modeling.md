# M-07.06 - Transit Node Modeling (Remove Edge Lag)

**Status:** ✅ Complete  
**Dependencies:** ✅ M-07.04 (Edge Semantics Contract)  
**Target:** Remove edge-level lag behavior and replace with explicit transit nodes.

---

## Overview

Edges must not own behavior (lag, queues, capacity). This milestone eliminates edge-level lag from templates and replaces it with explicit transit nodes (`service` by default, no queue), preserving determinism and making the model explainable.

### Strategic Context
- **Motivation:** Edge lag is confusing and violates the “no edge-level behavior” contract.
- **Impact:** Transit delays become visible, testable nodes with clear semantics.
- **Guardrail:** Any remaining edge lag is a warning and should be migrated.

## Scope

### In Scope ✅
1. Replace all edge lag usage in templates with explicit transit nodes.
2. Update invariants/tests to enforce no edge lag in active templates.
3. Update UI/tooltip text to remove lag phrasing where transit nodes exist.

### Out of Scope ❌
- Derived path latency (M-07.05).
- Dependency/resource constraints (separate epic).

## Requirements

### Functional Requirements

#### FR1: Transit nodes replace edge lag
**Acceptance Criteria:**
- [x] No active template uses edge `lag` for transit.
- [x] Transit nodes use explicit service semantics and own timing (queues only when the domain truly requires buffering).
- [x] Edge lag warnings are eliminated in templates after migration.

#### FR2: Engine enforcement
**Acceptance Criteria:**
- [x] Invariants flag edge lag as a warning (already in place).
- [x] Tests fail if lag remains in first-party templates.

#### FR3: UI clarity
**Acceptance Criteria:**
- [x] Edge tooltip no longer displays lag for migrated templates.
- [x] Transit nodes display their delay metrics in inspector/popup.

### Non-Functional Requirements

#### NFR1: Explainable timing
**Target:** Users can explain timing by inspecting nodes, not edges.
