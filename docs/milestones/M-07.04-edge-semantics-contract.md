# M-07.04 - Edge Semantics Contract (Engine-First)

**Status:** ✅ Complete  
**Dependencies:** ✅ M-07.01, ✅ M-07.02, ✅ M-07.03  
**Target:** Make edge metrics explicit and authoritative in the engine/API so clients never infer throughput vs effort semantics in the UI.

---

## Overview

The edge-time-bin epic introduced edge overlays and conservation warnings, but the **edge payload is still ambiguous** for retry and dependency load. This milestone makes the engine responsible for **edge semantics and edge metrics** so all consumers (UI, MCP, CLI) can interpret edges consistently.
It is a prerequisite for derived path latency (M-07.05), which relies on explicit edge metrics.

### Strategic Context
- **Motivation:** Throughput vs effort edges must be explicit to avoid UI inference and client divergence.
- **Impact:** API becomes the single source of truth for edge metrics and warnings.
- **Guardrail:** No client should derive retry load or attempts from node series; the engine must emit explicit edge metrics.

## Scope

### In Scope ✅
1. **Explicit edge metrics** in `/state_window` (per edge, per bin).
2. **Edge warnings keyed by edge id** (avoid parsing message strings in clients).
3. **Edge semantics contract** in `/graph` and schema.
4. **UI consumes explicit edge metrics** (no retry inference).

### Out of Scope ❌
- Transit modeling (edge lag removal) — see M-07.06.
- Path latency derivation — see M-07.05.
- Dependency/resource modeling — separate epic.

## Requirements

### Functional Requirements

#### FR1: Edge metrics payload
**Description:** `/state_window` includes explicit, per-edge metric series.

**Acceptance Criteria:**
- [ ] Throughput edges expose `flowVolume` (served flow only).
- [ ] Effort edges expose `attemptsVolume` (total attempt load).
- [ ] Failure/terminal edges expose `failuresVolume` (failed volume).
- [ ] Optional: emit `retryVolume` explicitly when available; otherwise it is derived as `attemptsVolume - flowVolume`.
- [ ] Metrics include `origin` metadata (`explicit` vs `derived`).
- [ ] Edge data is optional; when absent, `edges` is an empty array and `edgeWarnings` is an empty object.

#### FR2: Edge warnings keyed by edge id
**Description:** The engine emits `edgeWarnings` keyed by `edgeId`.

**Acceptance Criteria:**
- [ ] Edge warnings are emitted in `/state_window` as a keyed structure: `{ [edgeId]: StateWarning[] }`.
- [ ] UI uses `edgeWarnings` directly for badge coloring.
- [ ] No string parsing required to map warnings to edges.

#### FR3: Edge semantics contract
**Description:** `/graph` and schema explicitly define edge type and measure.

**Acceptance Criteria:**
- [ ] `edgeType` enumerates `throughput`, `effort`, `terminal`, `dependency`.
- [ ] `measure` clarifies which node series drives the edge (`served`, `attempts`, `failures`).
- [ ] Schemas updated and documented.

#### FR4: UI alignment
**Description:** UI draws badges/tooltips from explicit edge metrics only.

**Acceptance Criteria:**
- [ ] Retry load shown only when provided by engine edge metrics.
- [ ] Edge tooltip shows `flow`, `retries`, `total load` based on explicit series.
- [ ] No fallback inference from node metrics.

### Non-Functional Requirements

#### NFR1: Deterministic, engine-owned semantics
**Target:** All clients observe identical edge values for the same run.

#### NFR2: Backward compatibility
**Target:** Existing runs without edge metrics remain usable; UI degrades gracefully.
