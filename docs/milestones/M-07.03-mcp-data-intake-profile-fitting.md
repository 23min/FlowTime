# M-07.03 - MCP Data Intake and Profile Fitting

**Status:** Planned  
**Dependencies:** M-07.02 (MCP Modeling Draft Workflow)  
**Target:** Add data intake and input profile fitting tools to the MCP server.

---

## Overview

This milestone adds tools that let users provide input data (CSV or pasted tables), summarize the data, and fit input profiles that drive arrivals or service time distributions. The goal is to reduce modeling friction and allow users to bring real-world data into the chat workflow.

### Strategic Context
- **Motivation:** Many users have data but do not understand PMF terminology.
- **Impact:** Data-driven input profiles can be created through simple, guided steps.
- **Dependencies:** Requires draft workflow so profiles can be applied to templates.

---

## Scope

### In Scope
1. Data ingestion for CSV or pasted tables.
2. Series summarization (min, avg, peak, percentiles, periodicity).
3. Profile fitting from samples or summary statistics.
4. Mapping fitted profiles to template inputs.
5. Document pre-aggregated CSV format and column mapping guidance.

### Out of Scope
- Chart digitization from images.
- Advanced statistical model selection or auto-ML.
- Long-term storage or cataloging of profiles.
 - Raw timestamped series ingestion and binning (v1 uses pre-aggregated series only).

### Future Work (Optional)
- Move profile fitting into FlowTime.Sim for reuse across CLI/UI/API.
- Add optional chart-to-series tooling outside the MCP server.

## Requirements

### CSV Format (v1 - Pre-Aggregated Series)

The v1 ingestion flow accepts **pre-aggregated** data, aligned to the model grid. The CSV should include:
- `bin` (integer) — zero-based bin index.
- `value` (number) — value for that bin.

Example:
```
bin,value
0,120
1,150
2,180
3,160
```

Units, time range, and bin size are provided via metadata fields in the tool request (not embedded in the CSV).

### Functional Requirements

#### FR1: Series Ingestion
**Description:** Accept CSV or pasted tabular data and store it as a named series.

**Acceptance Criteria:**
- [ ] Series can be stored with metadata (units, source, time range).
- [ ] Invalid or malformed data returns clear errors.
- [ ] Documentation specifies a pre-aggregated CSV format (bin/value) and column mapping.

#### FR2: Series Summarization
**Description:** Provide a summary of the series to confirm it before fitting.

**Acceptance Criteria:**
- [ ] Summary includes min, avg, peak, and percentiles.
- [ ] Periodicity detection is reported when possible.

#### FR3: Profile Fitting
**Description:** Convert samples or summary statistics into an input profile.

**Acceptance Criteria:**
- [ ] Profiles can be produced from raw samples or summary stats.
- [ ] Fitted profiles are previewable before applying.

#### FR4: Mapping to Inputs
**Description:** Attach fitted profiles to specific template inputs.

**Acceptance Criteria:**
- [ ] Profiles can be applied to arrivals or service time inputs.
- [ ] Applied profiles are tracked with provenance metadata.

### Non-Functional Requirements

#### NFR1: Transparency
**Target:** Users can see how profiles were derived from data.
**Validation:** Each profile includes a summary of inputs and fitting method.

#### NFR2: Output Detail Levels
**Target:** The MCP tools support `detailLevel` output (`basic` or `expert`).
**Validation:** Each data-intake/fitting response can be returned with either a short summary or expanded diagnostics depending on `detailLevel`.
